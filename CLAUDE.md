# DevOps Defender (DD)

DevOps Defender is an open-source confidential computing platform that deploys unmodified Docker applications to hardware-verified Intel TDX (Trusted Domain Extensions) enclaves.

**Domain:** devopsdefender.com

## Zero-Secret Trust Model

The platform eliminates long-lived secrets through three complementary layers:

1. **Intel TDX** (trust root) ‚Äî hardware attestation proves each VM's identity and integrity via cryptographic quotes generated by the CPU. The control plane verifies these through Intel Trust Authority (ITA) before granting access.
2. **GitHub OIDC** (authentication) ‚Äî deployers authenticate using short-lived JWT tokens issued by GitHub Actions at deploy time. No API keys or credentials are stored as secrets.
3. **Cloudflare Tunnels** (networking) ‚Äî each agent gets an outbound-only tunnel for secure external access. No open ports, no TLS cert management, no firewall rules. Tunnel tokens are issued at registration and scoped to the agent.

Together: TDX proves the hardware is legit, GitHub OIDC proves the deployer is authorized, and Cloudflare provides the secure network path ‚Äî zero stored secrets end to end.

## Component Details

### agent/ (Rust)

A daemon that runs on each TDX VM node. It registers with the control plane via hardware attestation, establishes a Cloudflare tunnel, pulls and runs container workloads, and sends heartbeats.

**Binaries:** `dd-agent`
**Entry point:** `src/bin/dd-agent/main.rs`

**Three operational modes** (set via `DD_AGENT_MODE`):
- **agent** (default) ‚Äî register, attest, run workloads, heartbeat loop
- **control-plane** ‚Äî proxy mode, forwards config to `dd-cp` subprocess
- **measure** ‚Äî print TDX measurements and exit

**Key modules:**
- `src/bin/dd-agent/config.rs` ‚Äî config loading from `/etc/devopsdefender/agent.json` + env var overrides
- `src/bin/dd-agent/oci/mod.rs` ‚Äî container runtime interface via bollard (Docker/Podman)
- `src/attestation/tsm.rs` ‚Äî TDX quote generation/parsing via configfs-tsm
- `src/common/error.rs` ‚Äî shared error types

**Build system:** `build.rs` generates Rust structs from `openapi/control-plane.yaml` (code generation for API response types).

**Key env vars:** `DD_CONFIG`, `DD_AGENT_MODE`, `DD_CP_URL`, `DD_INTEL_API_KEY`, `DD_DATACENTER`, `DD_AGENT_SKIP_ATTESTATION`, `DD_PORT`

**Agent registration flow:**
1. GET `/api/v1/agents/challenge` ‚Üí receives nonce
2. Generate TDX quote embedding the nonce
3. POST `/api/v1/agents/register` with attestation token
4. Receive agent_id, tunnel_token, hostname
5. Start cloudflared tunnel, pull and run workload containers
6. POST heartbeat every 30s to `/api/v1/agents/{id}/heartbeat`

### control-plane/ (Rust + Axum)

Central management API for agent registration, deployment orchestration, health monitoring, and attestation verification.

**Binaries:** `dd-cp` (server), `dd-admin` (CLI utility for password hashing)
**Entry point:** `src/bin/dd-cp/main.rs`
**Database:** SQLite via sqlx with compile-time query checking

**Architecture layers:**
- `src/routes/` ‚Äî Axum HTTP handlers (health, agents, deploy, accounts, auth, admin, stats, ui)
- `src/services/` ‚Äî business logic (attestation, tunnel, nonce, github_oidc, health_monitor, gcp)
- `src/stores/` ‚Äî data access layer (agent, deployment, health, account, session, setting)
- `src/auth/` ‚Äî API key (Argon2, prefix `dd_live_`) and admin session (Argon2, prefix `dds_`) management
- `src/attestation/ita.rs` ‚Äî Intel Trust Authority JWT verification
- `src/api.rs` ‚Äî request/response DTOs
- `src/types.rs` ‚Äî enums (AgentStatus, DeploymentStatus, etc.)
- `src/config.rs` ‚Äî CpConfig loaded from env vars
- `src/state.rs` ‚Äî AppState shared across handlers

**Database schema** (defined in `migrations/0001_init.sql`):
- `agents` ‚Äî registered TDX nodes (status, hostname, tunnel_id, mrtd, tcb_status, etc.)
- `agent_control_credentials` ‚Äî SSH/control keys per agent
- `deployments` ‚Äî Docker Compose workload deployments per agent
- `app_health_checks` ‚Äî health check results
- `services` ‚Äî running containers
- `apps` / `app_versions` ‚Äî application catalog and deployment templates
- `accounts` ‚Äî API access accounts with hashed keys
- `admin_sessions` ‚Äî temporary admin auth tokens (24h expiry)
- `settings` ‚Äî key-value config store
- `trusted_mrtds` ‚Äî whitelisted TDX measurements

**Key API endpoints:**
- `GET /health` ‚Äî health check (boot_id, git_sha)
- `GET /api/v1/agents/challenge` ‚Äî issue registration nonce
- `POST /api/v1/agents/register` ‚Äî register with ITA attestation token
- `POST /api/v1/deploy` ‚Äî create deployment (finds available agent, supports dry-run)
- `GET/POST /api/v1/agents/{id}/checks` ‚Äî health check ingestion
- `POST /api/v1/accounts` ‚Äî create API account
- `POST /api/v1/auth/login` ‚Äî admin login ‚Üí session token
- `GET/POST /api/v1/admin/measurements` ‚Äî manage trusted MRTDs

**Key env vars:** `DD_CP_BIND_ADDR` (default 0.0.0.0:8080), `DD_CP_DATABASE_URL`, `DD_CP_ADMIN_PASSWORD`, `DD_CP_ITA_JWKS_URL`, `DD_CP_CF_API_TOKEN`, `DD_CP_CF_ACCOUNT_ID`, `DD_CP_CF_ZONE_ID`, `DD_CP_CF_DOMAIN` (default devopsdefender.com), `DD_ENV` (local/staging/production)

**Enforcement modes** (strict/warn/disabled): `DD_CP_TCB_ENFORCEMENT_MODE`, `DD_CP_RTMR_ENFORCEMENT_MODE`, `DD_CP_NONCE_ENFORCEMENT_MODE`

**Deployment:** Multi-stage Dockerfile (Rust builder ‚Üí debian bookworm-slim runtime with cloudflared).

**Infrastructure** (`infra/`): Ansible playbooks for GCP and baremetal deployments. Key playbooks:
- `gcp-control-plane-new.yml` ‚Äî launch TDX-enabled GCP VM for control plane
- `gcp-vm-fleet-new.yml` ‚Äî launch agent fleet
- `gcp-image-bake.yml` ‚Äî build VM images
- `baremetal-deploy.yml` ‚Äî bare metal deployment

### images/ (Packer)

Builds VM images for agent nodes with all dependencies pre-installed.

**Two image types:**
- `packer/baremetal-agent-image.pkr.hcl` ‚Äî QEMU/KVM image (Ubuntu 24.04, qcow2 output)
- `packer/gcp-agent-image.pkr.hcl` ‚Äî GCP Compute Engine image

**Provisioning script** (`packer/provision-agent-image.sh`) installs:
- dd-agent and dd-cp binaries
- Docker CE with buildx and containerd
- cloudflared (Cloudflare Tunnel client)
- Creates systemd services: `devopsdefender-agent.service` (enabled) and `devopsdefender-control-plane.service` (disabled)

### private-llm/ (Docker Compose)

Example application: a self-hosted LLM chat interface deployed to DevOps Defender.

**Services:**
- **Ollama** ‚Äî LLM inference engine
- **Caddy** ‚Äî reverse proxy routing `/api/*` to Ollama, serving static chat UI

**Two compose variants:**
- `docker-compose.yml` ‚Äî CPU mode, uses `smollm2:135m` model
- `docker-compose.h100.yml` ‚Äî GPU mode (NVIDIA H100), uses `qwen2.5:32b` model

**UI:** Single-page vanilla JS chat app (`ui/index.html`) with streaming responses, dark theme.
**Test:** `test.py` ‚Äî smoke test that validates model availability and inference.
**Deployment:** GitHub Actions workflow using OIDC authentication (no long-lived secrets).

### website/ (Static HTML)

Landing page for devopsdefender.com, deployed via GitHub Pages.
- `index.html` ‚Äî marketing page explaining the platform
- `style.css` ‚Äî design system (indigo/teal theme, responsive, dark mode support)
- `CNAME` ‚Äî GitHub Pages custom domain config

## Development Workflow & Branch Rules

### Branch Protection (Enforced)

- **`main` is protected.** All changes to `main` require a pull request with at least one approving review from a board member / repository owner. Direct pushes to `main` are blocked.
- **Stale reviews are dismissed** ‚Äî if you push new commits to a PR, previous approvals are invalidated and re-approval is required.

### Working on Changes

1. **Always work on a feature branch.** Create a branch from `main` (e.g., `feat/my-change`, `fix/bug-description`).
2. **Push your feature branch** and open a PR targeting `main`.
3. **Wait for board approval** before merging. Never merge your own PR without an approving review.

### Testing in Staging

- The **`staging` branch** auto-deploys to the staging environment (`app-staging.devopsdefender.com`).
- You can freely push to `staging` to test changes without approval.
- To test: merge or push your feature branch into `staging`. This triggers the staging deploy pipeline.
- The `staging` branch can be force-pushed or reset as needed ‚Äî it is not protected.

### Summary

| Branch    | Protection | Deploys to  | Approval Required |
|-----------|-----------|-------------|-------------------|
| `main`    | Yes       | Production-ready | Yes ‚Äî board review required |
| `staging` | No        | Staging     | No ‚Äî free to test |
| Feature   | No        | ‚Äî           | No ‚Äî work freely  |

```bash
# Control plane
DD_CP_DATABASE_URL=sqlite://dev.db DD_CP_ADMIN_PASSWORD=changeme cargo run --bin dd-cp

# Agent (skip attestation for non-TDX environments)
DD_AGENT_SKIP_ATTESTATION=true DD_CP_URL=http://localhost:8080 cargo run --bin dd-agent

```

## Key Concepts

- **Intel TDX**: Hardware-based trusted execution environment. VMs run in encrypted memory. The hardware generates attestation quotes proving the VM's identity and integrity.
- **MRTD**: Measurement of the TDX domain ‚Äî a hash representing the initial VM state. Trusted MRTDs are whitelisted in the control plane.
- **RTMR**: Runtime measurements (RTMR0-3) ‚Äî extend-only registers tracking runtime state changes.
- **ITA (Intel Trust Authority)**: External service that verifies TDX quotes and issues signed JWTs.
- **Cloudflare Tunnels**: Each agent gets a tunnel for secure external access without exposing ports.
- **GitHub OIDC**: Deployment authentication uses GitHub Actions' built-in OIDC tokens ‚Äî no stored secrets needed.
- **configfs-tsm**: Linux kernel interface (`/sys/kernel/config/tsm/report/`) for generating TDX quotes.

## Deployments & Operations

All infrastructure is managed through GitHub Actions. **Never SSH into hosts, run Ansible locally, or attempt manual fixes on VMs.** If staging or production is down, trigger the appropriate GitHub Actions workflow.

**Staging** (`staging-deploy.yml`) ‚Äî auto-deploys on push to `main`. Pipeline: build ‚Üí bake GCP agent image ‚Üí cleanup old VMs ‚Üí deploy via Ansible ‚Üí smoke check `app-staging.devopsdefender.com/health`.

**Production** (`production-deploy.yml`) ‚Äî manual trigger only (`workflow_dispatch`). Inputs: `num_tiny_agents`, `num_standard_agents`, `num_llm_agents`. Same pipeline as staging.

**Health check URLs:**
- Staging: `https://app-staging.devopsdefender.com/health`
- Production: `https://app.devopsdefender.com/health`

**To deploy or fix an environment**, use `gh workflow run` or the GitHub Actions UI:
```bash
# Trigger staging deploy
gh workflow run staging-deploy.yml --repo devopsdefender/control-plane

# Trigger production deploy
gh workflow run production-deploy.yml --repo devopsdefender/control-plane \
  -f num_tiny_agents=0 -f num_standard_agents=0 -f num_llm_agents=1
```
