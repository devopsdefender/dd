# DevOps Defender (DD)

DevOps Defender is an open-source confidential computing platform that deploys unmodified Docker applications to hardware-verified Intel TDX (Trusted Domain Extensions) enclaves.

**Domain:** devopsdefender.com

## Zero-Secret Trust Model

The platform eliminates long-lived secrets through three complementary layers:

1. **Intel TDX** (trust root) — hardware attestation proves each VM's identity and integrity via cryptographic quotes generated by the CPU. The control plane verifies these through Intel Trust Authority (ITA) before granting access.
2. **GitHub OIDC** (authentication) — deployers authenticate using short-lived JWT tokens issued by GitHub Actions at deploy time. No API keys or credentials are stored as secrets.
3. **Cloudflare Tunnels** (networking) — each agent gets an outbound-only tunnel for secure external access. No open ports, no TLS cert management, no firewall rules. Tunnel tokens are issued at registration and scoped to the agent.

Together: TDX proves the hardware is legit, GitHub OIDC proves the deployer is authorized, and Cloudflare provides the secure network path — zero stored secrets end to end.

## Component Details

### agent/ (Rust)

A daemon that runs on each TDX VM node. It registers with the control plane via hardware attestation, establishes a Cloudflare tunnel, pulls and runs container workloads, and sends heartbeats.

**Binaries:** `dd-agent`
**Entry point:** `src/bin/dd-agent/main.rs`

**Three operational modes** (set via `DD_AGENT_MODE`):
- **agent** (default) — register, attest, run workloads, heartbeat loop
- **control-plane** — proxy mode, forwards config to `dd-cp` subprocess
- **measure** — print TDX measurements and exit

**Key modules:**
- `src/bin/dd-agent/config.rs` — config loading from `/etc/devopsdefender/agent.json` + env var overrides
- `src/bin/dd-agent/oci/mod.rs` — container runtime interface via bollard (Docker/Podman)
- `src/attestation/tsm.rs` — TDX quote generation/parsing via configfs-tsm
- `src/common/error.rs` — shared error types

**Build system:** `build.rs` generates Rust structs from `openapi/control-plane.yaml` (code generation for API response types).

**Key env vars:** `DD_CONFIG`, `DD_AGENT_MODE`, `DD_CP_URL`, `DD_INTEL_API_KEY`, `DD_DATACENTER`, `DD_AGENT_SKIP_ATTESTATION`, `DD_PORT`

**Agent registration flow:**
1. GET `/api/v1/agents/challenge` → receives nonce
2. Generate TDX quote embedding the nonce
3. POST `/api/v1/agents/register` with attestation token
4. Receive agent_id, tunnel_token, hostname
5. Start cloudflared tunnel, pull and run workload containers
6. POST heartbeat every 30s to `/api/v1/agents/{id}/heartbeat`

### control-plane/ (Rust + Axum)

Central management API for agent registration, deployment orchestration, health monitoring, and attestation verification.

**Binaries:** `dd-cp` (server), `dd-admin` (CLI utility for password hashing)
**Entry point:** `src/bin/dd-cp/main.rs`
**Database:** SQLite via sqlx with compile-time query checking

**Architecture layers:**
- `src/routes/` — Axum HTTP handlers (health, agents, deploy, accounts, auth, admin, stats, ui)
- `src/services/` — business logic (attestation, tunnel, nonce, github_oidc, health_monitor, gcp)
- `src/stores/` — data access layer (agent, deployment, health, account, session, setting)
- `src/auth/` — API key (Argon2, prefix `dd_live_`) and admin session (Argon2, prefix `dds_`) management
- `src/attestation/ita.rs` — Intel Trust Authority JWT verification
- `src/api.rs` — request/response DTOs
- `src/types.rs` — enums (AgentStatus, DeploymentStatus, etc.)
- `src/config.rs` — CpConfig loaded from env vars
- `src/state.rs` — AppState shared across handlers

**Database schema** (defined in `migrations/0001_init.sql`):
- `agents` — registered TDX nodes (status, hostname, tunnel_id, mrtd, tcb_status, etc.)
- `agent_control_credentials` — SSH/control keys per agent
- `deployments` — Docker Compose workload deployments per agent
- `app_health_checks` — health check results
- `services` — running containers
- `apps` / `app_versions` — application catalog and deployment templates
- `accounts` — API access accounts with hashed keys
- `admin_sessions` — temporary admin auth tokens (24h expiry)
- `settings` — key-value config store
- `trusted_mrtds` — whitelisted TDX measurements

**Key API endpoints:**
- `GET /health` — health check (boot_id, git_sha)
- `GET /api/v1/agents/challenge` — issue registration nonce
- `POST /api/v1/agents/register` — register with ITA attestation token
- `POST /api/v1/deploy` — create deployment (finds available agent, supports dry-run)
- `GET/POST /api/v1/agents/{id}/checks` — health check ingestion
- `POST /api/v1/accounts` — create API account
- `POST /api/v1/auth/login` — admin login → session token
- `GET/POST /api/v1/admin/measurements` — manage trusted MRTDs

**Key env vars:** `DD_CP_BIND_ADDR` (default 0.0.0.0:8080), `DD_CP_DATABASE_URL`, `DD_CP_ADMIN_PASSWORD`, `DD_CP_ITA_JWKS_URL`, `DD_CP_CF_API_TOKEN`, `DD_CP_CF_ACCOUNT_ID`, `DD_CP_CF_ZONE_ID`, `DD_CP_CF_DOMAIN` (default devopsdefender.com), `DD_ENV` (local/staging/production)

**Enforcement modes** (strict/warn/disabled): `DD_CP_TCB_ENFORCEMENT_MODE`, `DD_CP_RTMR_ENFORCEMENT_MODE`, `DD_CP_NONCE_ENFORCEMENT_MODE`

**Deployment:** Multi-stage Dockerfile (Rust builder → debian bookworm-slim runtime with cloudflared).

**Infrastructure** (`infra/`): Ansible playbooks for GCP and baremetal deployments. Key playbooks:
- `gcp-control-plane-new.yml` — launch TDX-enabled GCP VM for control plane
- `gcp-vm-fleet-new.yml` — launch agent fleet
- `gcp-image-bake.yml` — build VM images
- `baremetal-deploy.yml` — bare metal deployment

### images/ (Packer)

Builds VM images for agent nodes with all dependencies pre-installed.

**Two image types:**
- `packer/baremetal-agent-image.pkr.hcl` — QEMU/KVM image (Ubuntu 24.04, qcow2 output)
- `packer/gcp-agent-image.pkr.hcl` — GCP Compute Engine image

**Provisioning script** (`packer/provision-agent-image.sh`) installs:
- dd-agent and dd-cp binaries
- Docker CE with buildx and containerd
- cloudflared (Cloudflare Tunnel client)
- Creates systemd services: `devopsdefender-agent.service` (enabled) and `devopsdefender-control-plane.service` (disabled)

### private-llm/ (Docker Compose)

Example application: a self-hosted LLM chat interface deployed to DevOps Defender.

**Services:**
- **Ollama** — LLM inference engine
- **Caddy** — reverse proxy routing `/api/*` to Ollama, serving static chat UI

**Two compose variants:**
- `docker-compose.yml` — CPU mode, uses `smollm2:135m` model
- `docker-compose.h100.yml` — GPU mode (NVIDIA H100), uses `qwen2.5:32b` model

**UI:** Single-page vanilla JS chat app (`ui/index.html`) with streaming responses, dark theme.
**Test:** `test.py` — smoke test that validates model availability and inference.
**Deployment:** GitHub Actions workflow using OIDC authentication (no long-lived secrets).

### website/ (Static HTML)

Landing page for devopsdefender.com, deployed via GitHub Pages.
- `index.html` — marketing page explaining the platform
- `style.css` — design system (indigo/teal theme, responsive, dark mode support)
- `CNAME` — GitHub Pages custom domain config

## Build & Development

### Rust Projects (agent/, control-plane/)

```bash
# Build
cargo build

# Run tests
cargo test

# Format check
cargo fmt --check

# Lint
cargo clippy --all-targets
```

CI enforces: `RUSTFLAGS="-Dwarnings"` (warnings are errors).

Both projects use Rust 2021 edition. The control-plane uses sqlx with compile-time checked queries against SQLite.

### Running Locally

```bash
# Control plane
DD_CP_DATABASE_URL=sqlite://dev.db DD_CP_ADMIN_PASSWORD=changeme cargo run --bin dd-cp

# Agent (skip attestation for non-TDX environments)
DD_AGENT_SKIP_ATTESTATION=true DD_CP_URL=http://localhost:8080 cargo run --bin dd-agent

# Generate admin password hash
cargo run --bin dd-admin -- hash-password

# Private LLM
cd private-llm && docker compose up
```

## Key Concepts

- **Intel TDX**: Hardware-based trusted execution environment. VMs run in encrypted memory. The hardware generates attestation quotes proving the VM's identity and integrity.
- **MRTD**: Measurement of the TDX domain — a hash representing the initial VM state. Trusted MRTDs are whitelisted in the control plane.
- **RTMR**: Runtime measurements (RTMR0-3) — extend-only registers tracking runtime state changes.
- **ITA (Intel Trust Authority)**: External service that verifies TDX quotes and issues signed JWTs.
- **Cloudflare Tunnels**: Each agent gets a tunnel for secure external access without exposing ports.
- **GitHub OIDC**: Deployment authentication uses GitHub Actions' built-in OIDC tokens — no stored secrets needed.
- **configfs-tsm**: Linux kernel interface (`/sys/kernel/config/tsm/report/`) for generating TDX quotes.

## Deployments & Operations

All infrastructure is managed through GitHub Actions. **Never SSH into hosts, run Ansible locally, or attempt manual fixes on VMs.** If staging or production is down, trigger the appropriate GitHub Actions workflow.

**Staging** (`staging-deploy.yml`) — auto-deploys on push to `main`. Pipeline: build → bake GCP agent image → cleanup old VMs → deploy via Ansible → smoke check `app-staging.devopsdefender.com/health`.

**Production** (`production-deploy.yml`) — manual trigger only (`workflow_dispatch`). Inputs: `num_tiny_agents`, `num_standard_agents`, `num_llm_agents`. Same pipeline as staging.

**Health check URLs:**
- Staging: `https://app-staging.devopsdefender.com/health`
- Production: `https://app.devopsdefender.com/health`

**To deploy or fix an environment**, use `gh workflow run` or the GitHub Actions UI:
```bash
# Trigger staging deploy
gh workflow run staging-deploy.yml --repo devopsdefender/control-plane

# Trigger production deploy
gh workflow run production-deploy.yml --repo devopsdefender/control-plane \
  -f num_tiny_agents=0 -f num_standard_agents=0 -f num_llm_agents=1
```
