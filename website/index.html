How It Works

Four steps from code to enclave

DevOps Defender handles attestation, networking, and orchestration so you can focus on your application.

Register App

Create an application entry via the API. Define the container image, health check, and resource requirements.

Publish Version

Push a new app version from GitHub Actions. The OIDC token proves ownership — no API key needed.

Measure Enclave

The control plane records TDX measurements (MRTD, RTMR) for the target VM. Only trusted measurements are accepted.

Deploy

An available TDX agent picks up the deployment, pulls the image, starts the container, and opens a Cloudflare tunnel.

Defense in Depth

Six layers of verification

Security isn't one feature — it's the interaction between hardware, firmware, kernel, and application-level checks.

Nonce Challenge Protocol

Every registration begins with a fresh nonce embedded in the TDX quote, preventing replay attacks.

TDX Measurements

MRTD and RTMR[0–3] registers capture the full boot chain — firmware, kernel, root filesystem, and application layer.

dm-verity Root FS

The agent VM root filesystem is integrity-protected. Any block-level tampering is detected at read time.

TCB Enforcement

Trusted Computing Base status is verified against Intel advisories. Out-of-date firmware is rejected.

Cloudflare Tunnel Isolation

Agents make outbound-only connections. No inbound ports, no public IPs, no attack surface on the network layer.

Continuous Health Monitoring

Agents heartbeat every 30 seconds. Missed heartbeats trigger re-attestation before any new workload is scheduled.

GitHub OIDC

Deploy without secrets

GitHub Actions issues short-lived OIDC tokens. The control plane verifies them directly — no API keys, no shared passwords, no credential rotation.

  1. GitHub Actions mints a JWT bound to the repository and workflow
  2. The control plane verifies the token against GitHub's JWKS endpoint
  3. Repository ownership is matched to the registered application
  4. Deployment is created and assigned to an attested TDX agent
# .github/workflows/deploy.yml
name: Deploy to TDX
on:
  push:
    branches: [main]

permissions:
  id-token: write

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Get OIDC Token
        id: oidc
        run: |
          TOKEN=$(curl -s -H \
            "Authorization: bearer $ACTIONS_ID_TOKEN_REQUEST_TOKEN" \
            "$ACTIONS_ID_TOKEN_REQUEST_URL&audience=devopsdefender" \
            | jq -r '.value')
          echo "token=$TOKEN" >> "$GITHUB_OUTPUT"

      - name: Deploy
        run: |
          curl -X POST \
            https://app.devopsdefender.com/api/v1/deploy \
            -H "Authorization: Bearer ${{ steps.oidc.outputs.token }}" \
            -H "Content-Type: application/json" \
            -d '{"app_name": "my-app", "version": "latest"}'

Quick Start

Up and running in three steps

1. Create an account and register your app

curl -X POST https://app.devopsdefender.com/api/v1/accounts \
  -H "Content-Type: application/json" \
  -d '{"name": "my-org", "type": "deployer"}'

curl -X POST https://app.devopsdefender.com/api/v1/apps \
  -H "Authorization: Bearer dd_live_..." \
  -H "Content-Type: application/json" \
  -d '{
    "name": "my-app",
    "image": "ghcr.io/my-org/my-app:latest",
    "github_repo": "my-org/my-app",
    "health_check_path": "/health"
  }'

2. Add the GitHub Action to your repository

Copy the workflow above into .github/workflows/deploy.yml and push to main.

3. Verify the deployment

# Check deployment status
curl https://app.devopsdefender.com/api/v1/deployments \
  -H "Authorization: Bearer dd_live_..."

# Verify attestation
curl https://app.devopsdefender.com/api/v1/agents \
  -H "Authorization: Bearer dd_live_..." \
  | jq '.[].attestation'
