How It Works

From code commit to running in a hardware-secured enclave — four steps, no magic.

1

Register Your App

Tell the control plane about your app: the Docker image, and which GitHub organization is allowed to deploy it. That's it.

2

Push Your Image

Push your container image to any registry. DevOps Defender pins the exact content digest so only that version can run — no surprises.

3

Hardware Verifies Everything

The Intel CPU generates a signed proof that your VM booted with the right firmware, kernel, and filesystem — before your container ever starts.

4

Ship It — Securely

Your container launches inside the verified enclave. The control plane continuously monitors each agent; anything that fails attestation gets quarantined automatically.

What Makes It Secure

Intel TDX protects entire virtual machines in hardware — your hypervisor, cloud provider, and even your own ops team can't read workload memory.

Memory Nobody Else Can Read

All VM memory is encrypted in hardware. The hypervisor, other tenants, and even physical memory probes can't access your data — only the CPU can decrypt it.

  • Per-VM encryption keys generated inside the CPU
  • Keys never exposed to any software layer
  • Protection against memory replay and tampering
  • Full VM isolation, not just process-level sandboxing

Proof the Right Code Is Running

Before any workload gets access to secrets, the Intel CPU issues a signed attestation quote — cryptographic proof of exactly what software booted and what's running inside.

  • Initial firmware configuration measured at VM start
  • Firmware and kernel boot chain measured in sequence
  • Root filesystem integrity verified before mount
  • Application container measured at launch

No Code Changes Required

Unlike older enclave tech that required rewriting apps with a special SDK, TDX protects entire VMs. Your existing Docker containers run as-is inside a confidential environment.

  • Standard OCI images, any base image, any language
  • No SDK or library dependencies to add
  • Full Linux environment inside the enclave
  • Rootless container runtime for defense-in-depth

Tamper-Proof Boot Chain

Every component from firmware to your app is measured into hardware registers before it runs. An unforgeable chain of trust from CPU silicon to container — you can verify it yourself.

  • Firmware measured by the CPU at VM launch
  • Kernel and initramfs measured before execution
  • Read-only filesystem with integrity verification
  • Application identity locked in at container start

Defense in Depth

Layered security so that no single failure — hardware, software, or human — can compromise your workloads.

No Stale Attestations

Every verification challenge is unique and time-bound. An attacker can't replay an old proof — each attestation is fresh or it's rejected.

Tamper-Evident Filesystem

The agent VM boots with a read-only, integrity-verified filesystem. Any modification causes an immediate I/O error — persistent rootkits have nowhere to hide.

Pinned Software Stack

Firmware, kernel, and runtime versions are locked. Agents running anything unexpected are rejected before they can touch a workload.

Continuous Health Checks

Agents continuously re-attest to the control plane. If anything drifts or goes silent, that agent is quarantined from the fleet automatically.

No Open Ports

Agents reach out through an encrypted tunnel — no inbound firewall rules, no exposed services. Your workloads are unreachable from the open internet by default.

No Shared Secrets

Authorization is tied to your GitHub organization identity — no API keys to rotate, no credentials to leak. If your GitHub org can push, it can deploy. That's it.

Deploy from GitHub Actions — No Credentials Needed

GitHub vouches for your identity. DevOps Defender trusts GitHub. You never touch a secret.

1

GitHub Proves Who You Are

When your workflow runs, GitHub issues a signed token saying: this deployment came from this repo, this org, this branch. No passwords, no API keys — GitHub's identity is the credential.

2

Control Plane Checks the Signature

DevOps Defender verifies the token signature directly against GitHub's public keys. If it's genuine and the org matches, the deploy is authorized — no lookup tables, no stored secrets.

3

Only Your Agents Receive the Workload

The control plane routes the deployment only to agents registered under your organization. Other orgs on the same platform can't touch your workloads — and vice versa.

4

Container Launches, Nothing to Clean Up

Your app starts inside the verified enclave. No secrets were created, stored, or rotated. The entire auth flow is stateless — there's nothing to breach after the fact.

.github/workflows/deploy.yml YAML 
name: Deploy to DevOps Defender
on:
  push:
    branches: [main]

permissions:
  id-token: write
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Deploy to TDX enclave
        uses: devopsdefender/deploy-action@v1
        with:
          control-plane: https://devopsdefender.com
          app-name: my-application
          image: ghcr.io/myorg/myapp:${{ github.sha }}

Quick Start

Get your first workload running in a hardware-secured enclave in minutes — on infra you own.

Step 1

Register Your App

Tell your self-hosted control plane about the app: the container image and which GitHub organization owns it. One API call.

Terminal 
# Register a new application
curl -X POST https://devopsdefender.com/api/v1/apps \
  -H "Content-Type: application/json" \
  -d '{
    "name": "my-application",
    "image": "ghcr.io/myorg/myapp:latest",
    "owner": "github:org/myorg"
  }'
Step 2

Add the GitHub Action

Drop the deploy action into your workflow. It handles everything — no credentials to configure, no secrets to store in GitHub.

.github/workflows/deploy.yml 
permissions:
  id-token: write

steps:
  - name: Deploy
    uses: devopsdefender/deploy-action@v1
    with:
      control-plane: https://devopsdefender.com
      app-name: my-application
      image: ghcr.io/myorg/myapp:${{ github.sha }}
Step 3

Verify It Yourself

Confirm your agent is registered and the workload is live inside a verified enclave. You can inspect every attestation — it's your infrastructure.

Terminal 
# List registered agents
curl https://devopsdefender.com/api/v1/agents | jq

# Check agent attestation status
curl https://devopsdefender.com/api/v1/agents/{id} | jq '.registration_state'

# View agent logs
curl https://devopsdefender.com/api/v1/agents/{id}/logs
