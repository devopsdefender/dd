What “sealed” means.

Intel TDX seals the guest’s memory against the host. That part is table stakes: every confidential-compute platform claims it.

DD adds a stronger property on top: sealed against the operator too. A confidential-mode agent boots with DD_CONFIDENTIAL=true, which means /deploy, /exec, and /owner are not registered as routes. Nobody — not the customer, not DD ops, not the bot — has a channel to mutate the running workload post-boot. Logs and attestation stay open. The TDX quote measures the boot config, so the absence of mutation channels is visible from outside.

Without confidential mode

  • POST /deploy · live workload swap
  • POST /exec · arbitrary command run
  • ttyd shell · interactive access
  • POST /owner · reassign tenant

All four channels exist. Trust = trust the operator’s policy not to push them.

Confidential mode

  • POST /deploy · 404
  • POST /exec · 404
  • ttyd · not in workload
  • POST /owner · 404

Zero channels. Trust = verify the TDX quote covers a config disk that omits them.

Three steps to a sealed oracle.

  1. 01

    Publish a workload.json

    Public GitHub repo, file at root. The bot will fetch it at a specified commit and deploy as-is. No build step inside the enclave; binaries come from your own GitHub release assets via github_release entries. Same shape DD’s existing apps already use.

  "cmd": ["/var/lib/easyenclave/bin/my-oracle"],
  "expose": { "hostname_label": "oracle", "port": 8080 }
}
  2. 02

    Pay or run your own operator

    Pay BTC at the canonical operator (satsforcompute), or fork the example operator and run your own bot against your own DD fleet. Either way, the bot creates a GitHub-issue claim, watches mempool.space for payment, then deploys the workload onto a fresh agent in confidential mode.

  3. 03

    Verify the seal yourself

    Pull the agent’s /health JSON. Check the TDX quote was Intel-signed. Confirm taint_reasons contains exactly ["customer_workload_deployed"] — no exec, no shell, no owner-set. The absence of those channels is the proof, and it’s cryptographic.

Verifying the seal.

There’s no “DD trust me” step. Three commands against any agent’s public hostname: 

# 1. Fetch the TDX quote (Noise pre-handshake bundle on /health).
curl -fsSL https://<agent>.devopsdefender.com/health \
  | jq '.noise.quote_b64, .noise.pubkey_hex'

# 2. Verify the quote with Intel Trust Authority (or any TDX verifier).
#    The quote's report_data binds the agent's Noise pubkey,
#    and the MRTD is the measured boot.

# 3. Confirm the running workload is sealed.
curl -fsSL https://<agent>.devopsdefender.com/health \
  | jq '{ confidential_mode, taint_reasons }'

# Expected for a sealed oracle:
#   { "confidential_mode": true,
#     "taint_reasons": ["customer_workload_deployed"] }

The /health endpoint is CF-Access-bypassed and public. Anyone can verify any agent at any time without operator cooperation.

Trust state machine.

Taint is a set of reasons, not a boolean. Each reason points to a concrete mechanism that gave somebody influence over the node. A sealed oracle’s set is small and known.

pristine Booted from a known image. taint_reasons = []. No customer has had any authority since boot.
tainted One or more reasons in the set:
customer_workload_deployed · a /deploy succeeded since boot
customer_owner_enabled · /owner set a non-fleet tenant
arbitrary_exec_enabled · node booted with /deploy + /exec registered
interactive_shell_enabled · ttyd or equivalent in the running workload
safe_mode On reboot, the runtime owner clears and the node returns to bot/DD control. May still be tainted — tainted nodes are rebuilt before reassignment.

A sealed oracle ships with exactly {customer_workload_deployed} in the set. The absence of arbitrary_exec_enabled and interactive_shell_enabled is the cryptographic signal that the operator has no tamper channel.
